# FloraOS Base Rootfs Manifest

Every package/binary in the base image, and why it's there. Nothing gets added
without a line here. If you can't justify it in one line, it doesn't belong
in the base.

## Build-time only (not present in the final image)

| Component | Reason |
|---|---|
| this build host's own gcc/binutils | compiles every package from source. Every FloraOS binary is still built from pristine upstream source (nothing copied from Arch/Artix) -- we just use the compiler already on this machine to do the compiling, rather than *also* bootstrapping our own compiler from scratch first (a deliberate scope call: full LFS-style cross-toolchain bootstrapping adds hours and much more that can break, for no benefit beyond purity) |
| sanitized linux kernel headers | `make headers_install` from the linux-lts source, needed so glibc builds against the right kernel ABI |

## Runtime (shipped in the base rootfs)

| Package | Reason |
|---|---|
| linux-lts | kernel; LTS branch means fewer breaking changes to track on a from-scratch distro (see ARCHITECTURE.md) |
| glibc | libc; standard pairing with GNU userland, which the target spec mandates. `memusagestat` pruned post-install (needs libgd, which FloraOS doesn't ship) |
| ncurses | terminal capabilities (terminfo) library — bash/readline links against libncursesw.so.6 dynamically; without shipping it ourselves bash can't even load. Built with `--with-versioned-syms`: without it, every single invocation of anything linking against it warned "no version information available" (bash etc. are built against this build host's own *versioned* ncurses, so they request a version node ours didn't define) — cosmetic, but found on every line of a real interactive boot |
| bash | required default shell |
| coreutils | required GNU userland (ls, cp, mv, cat, ...) |
| util-linux | required GNU userland (mount, fdisk, losetup, ...) |
| e2fsprogs | mkfs.ext4/fsck.ext4 — minimum filesystem tooling to build and check the rootfs. `fsck.cramfs`/`mkfs.cramfs` pruned post-install (legacy filesystem FloraOS doesn't use, needs libz, which FloraOS doesn't ship) |
| sysvinit | PID1 — OpenRC is a runlevel/dependency manager, not a PID1 implementation itself, and needs one of sysvinit/busybox-init as a companion. sysvinit is the traditional OpenRC pairing (pre-systemd Gentoo/Arch/etc.) and, unlike busybox, isn't a single monolithic binary, keeping with the GNU-userland-only rule |
| openrc | runlevel/service manager, started by sysvinit; explicitly no systemd |
| dhcpcd | DHCP client; base networking as specified in the target spec |
| iproute2 | `ip` command for manual interface/route configuration when dhcpcd isn't enough |
| libmd | dhcpcd links against this (BSD message-digest routines) with no configure flag to avoid it, unlike most other auto-detected-optional-lib gaps found here |
| attr, acl | gnulib-based tools (sed, and potentially others later) auto-detect and link against libattr/libacl if present on whatever machine builds them, regardless of whether their own configure was told to care about xattrs/ACLs. Shipping these once closes that class of gap instead of chasing it per-package |
| grep | fau's own JSON parsing and package lookup logic uses grep — without it fau is broken *inside the running OS* (it only worked at build time because the build host has it) |
| sed | fau's JSON read/write functions use sed — same reasoning as grep |
| gawk | fau's repo/desc field parsing uses awk — same reasoning as grep |
| findutils | fau's install (app) bin-entrypoint detection uses find; dependency-list parsing uses xargs — same reasoning as grep |
| tar | fau's own package format (.fau.tar.zst) is a tar archive; fau extracts/builds these with tar — same reasoning as grep |
| zstd | fau's package format is .fau.tar.zst — fau can't extract or build any package without it — same reasoning as grep |
| rsync | fau's system-package installs merge via `rsync -aK` (needed to merge into the merged-/usr symlinks without replacing them) — same reasoning as grep |
| fau | FloraOS's own system manager (see tools/fau) — installs from the FloraOS package repo and owns the `system.json` reproducibility manifest natively; package management, `fau backup`'s btrfs snapshots, and `fau service-*`'s OpenRC front end are what exist today, with the intent of growing into managing the rest of the running system too. (Yes, fau needs the row above to function — a small bootstrapping irony worth naming rather than leaving implicit) |
| procps-ng | `sysctl` — OpenRC's `etc/init.d/sysctl` service (`sysctl --system`) failed non-fatally without it; the rest of the suite (ps/top/kill/free/watch/...) comes along for free from the same source tree. Only upstream package here without a generated `configure` in its release artifact (raw gitlab source archive) — needs `autoreconf` on the build host |
| hostname | `hostname`/`dnsdomainname` — OpenRC's `etc/init.d/hostname` service failed non-fatally without it. Deliberately the small standalone package (Debian's `hostname.c` + Makefile), not inetutils, which bundles telnet/ftp/rsh/talk/etc for the one command actually needed here |
| kbd | `loadkeys`/`dumpkeys`/`setfont` — OpenRC's `etc/init.d/keymaps` service failed non-fatally without it. Built with vlock/zlib/bzip2/lzma/xkb off (PAM, and libs FloraOS doesn't ship); zstd left on since fau already needs libzstd. loadkeys shells out to `gzip` to decompress `.gz` keymaps — see the `gzip` row below, which now closes that gap |
| gzip | `loadkeys`/`dumpkeys` (kbd, above) shell out to `gzip` to decompress `.gz`-compressed keymaps/fonts, falling back to their own internal decompression (with stderr noise) when it's missing — FloraOS didn't ship it. Plain autotools build, no optional deps |
| libxcrypt | glibc itself dropped `crypt()`/`crypt.h` a few versions back — floralogin (see below) needs it to verify `/etc/shadow` hashes. Built with `--enable-obsolete-api=glibc` for the traditional crypt(3) ABI (SONAME `libcrypt.so.1`) |
| mbedtls | TLS backend for curl (below) — the pacman mirrors are HTTPS-only. Picked over OpenSSL: a plain-Makefile build (no cmake, which this project uses nowhere else) and a smaller footprint. Only the shared libs + headers are staged (`make lib`, not `make install`, which unconditionally also builds mbedTLS's own example/test/fuzz programs) |
| curl | fau's alpm (Arch/Artix repo) fallback needs an HTTP client to fetch anything once running inside a booted FloraOS system (no pacman there to shell out to) — confirmed by literally running `fau install` after boot: "curl: command not found". Trimmed to HTTP/HTTPS only (no FTP/telnet/gopher/mqtt/etc, no libpsl, no libidn2, no nghttp2) — none of that is needed to fetch from a fixed set of mirror hostnames |
| ca-certificates | curl needs a trust store to validate the mirrors' HTTPS certs. Not a compiled package — Mozilla's root CA list, maintained and republished by the curl project specifically for systems without their own CA infrastructure. Fetched directly in build-rootfs.sh (a plain data file, not a tarball) rather than through the recipe pipeline |
| eudev | systemd-free udev fork — libinput (and mesa/wlroots, fetched later via `fau install <wm>`) hard-require libudev at build and run time, no supported fallback exists upstream. Gives FloraOS `/dev/input/*`/`/dev/dri/*` device nodes with correct perms plus hotplug uevents. Built with `--disable-blkid --disable-selinux --disable-kmod` — none of the three are needed for device nodes/hotplug, and all three degrade gracefully instead of failing the build. `--disable-kmod` means no module-autoload capability, so `linux-lts` now builds a small set of display/input drivers straight into vmlinuz instead (see below) rather than adding a second new build-host package (kmod) just for module loading. Adds exactly one new build-host requirement: `gperf` (configure.ac requires it unconditionally, confirmed directly against the real 3.2.14 tarball) |

## FloraOS-authored (not a fetched upstream package, compiled directly)

| Tool | Reason |
|---|---|
| floralogin (`tools/floralogin`) | FloraOS's own ~100-line, from-scratch, PAM-free login — util-linux's own login unconditionally requires PAM to build at all (no fallback exists upstream), and PAM isn't part of FloraOS. Verifies against `/etc/shadow` via crypt(3)/libxcrypt, execs the shell on success. `/etc/inittab` runs it via `agetty --skip-login --login-program`, same as `fau` itself, not tracked in fau's own package manifest since it's compiled straight from FloraOS's own source, not fetched from anywhere (see build-rootfs.sh). Now also creates/chowns `/run/user/<uid>` (0700) and exports `XDG_RUNTIME_DIR` before exec'ing the shell — no session manager exists to do this the usual way, but any Wayland compositor hard-requires it at startup |
| fauelf (`tools/fauelf`) | FloraOS's own small from-scratch ELF64 tool that rewrites an absolute-path `DT_NEEDED` entry (e.g. `/usr/lib/lua/5.1/lpeg.so`, found via neovim's lua51-lpeg dependency) to its bare basename in place. Needed because `fau`'s alpm (Arch/Artix repo) fallback isolates each app under its own directory: an absolute `DT_NEEDED` bypasses `LD_LIBRARY_PATH`/RPATH entirely (the dynamic linker only consults those for a bare soname), so it fails to load even though the dependency is correctly bundled right there — confirmed in a real chroot of the built rootfs, not just by reading the code. Compiled once (no FloraOS-specific headers/libs needed) and used both by `fau` itself (`app_install_one_alpm`, see tools/fau/fau) and directly by build-rootfs.sh, since it needs to run both at ISO-build time and inside an already-booted system |
| floraseat (`tools/floraseat`) | FloraOS's own seat-management daemon — real seatd is meson/ninja-only upstream, which this project otherwise avoids everywhere (see mbedtls's plain-Makefile pick over OpenSSL). Speaks the real seatd wire protocol (verified directly against upstream kennylevinsen/seatd source, not reconstructed from memory) so that precompiled wlroots/libseat fetched later via `fau install <wm>` talks to it unmodified — same "reimplement the protocol/data format, not the codebase" idea as fau's own alpm fallback. Single seat0, non-VT-bound (no VT-switch support yet, see docs/TODO.md), DRM/evdev/hidraw device allowlist only, socket-permission-gated (`/run/seatd.sock`, 0660 root:seat) same as real seatd. Started from `/etc/inittab` (respawn, it doesn't self-daemonize) rather than an openrc runlevel, same reasoning as dhcpcd's own inittab entry |
| florauser (`tools/florauser`) | FloraOS's own from-scratch `useradd`/`passwd`/`groupadd` equivalent — same reasoning as floralogin: shadow-utils expects a PAM-shaped world FloraOS doesn't have. Root was previously the *only* login that could ever exist; `florauser add <name> [groups]` allocates a fresh uid/gid (>=1000), a user-private group, a locked shadow entry, and a home dir, `florauser passwd <name>` hashes via the same libxcrypt (`crypt_gensalt`/`crypt_r`) floralogin already links against, and `florauser addtogroup`/`groupadd` cover joining e.g. the `seat` group floraseat's socket is gated on. Reads/writes `/etc/{passwd,group,shadow}` directly (own colon-field parser, not NSS/getpwnam) so it doesn't depend on an `/etc/nsswitch.conf` FloraOS never ships; every command was exercised end-to-end against scratch copies of those three files in this project's own sandbox (uid/gid allocation, supplementary groups, idempotent re-add, password hashing producing a real yescrypt hash, and the expected rejections) — not just compiled. Not yet verified: an actual root-privileged run against a live rootfs or a real login as an florauser-created account (no root/QEMU access available here) |
| florainstall (`tools/florainstall`) | FloraOS's own TUI disk installer, run manually from the live shell (not wired into boot). ncurses/menu-driven (the ncurses build above already produces `libmenuw`/`libformw` alongside `libncursesw`, so this adds no new package). Partitions the target disk (MBR, one bootable Linux partition — deliberately not GPT+"BIOS boot partition", which needs a partition-type GUID this project has no primary source to verify), formats btrfs with an `@` subvolume as the actual install target (a sibling `@snapshots` holds `fau backup`'s own snapshots later, see below and ARCHITECTURE.md), then `rsync`s the live system's own already-running "/" into `@` — there is no separate installer payload, the booted image already *is* the full OS. Neither btrfs-progs nor grub are base packages (see the two rows below) — both are fetched at install time via `fau`'s own alpm fallback, btrfs-progs onto the live system itself (it has to run before the target disk is even mounted) and grub straight into the target, then `grub-install` runs inside a `chroot` into the target. `/boot/grub/grub.cfg` itself is generated by `floragrub-cfg` (see below), not hand-written here. Account setup (root password, one optional extra user) execs the real `florauser` inside that same chroot with the terminal inherited, so its interactive password prompt runs directly against the target's own `/etc/shadow` — florainstall itself never touches a plaintext password. Boot-tested end-to-end for real in QEMU/KVM by `scripts/test-install.sh` (install → boot the installed disk → `fau backup` → `grub-reboot` into it → `fau backup-restore` → reboot); see ARCHITECTURE.md's fau-backup section for the real bugs that test caught and fixed. Still no UEFI support (BIOS/MBR only) — see docs/TODO.md |
| floragrub-cfg (`tools/floragrub-cfg`) | FloraOS's own small bash tool (not compiled — same "portable script that must ship in the running OS" reasoning as `fau` itself, see build-rootfs.sh) that writes `/boot/grub/grub.cfg`: one menuentry for the `@` subvolume plus one per `fau backup` snapshot found under `@snapshots`. Used by both `florainstall` (initial install) and `fau backup`/`backup-remove`/`backup-restore` (regenerating after a snapshot is added, removed, or promoted), so the menuentry format and subvolume-path logic exist in exactly one place rather than being duplicated across florainstall.c's C and fau's bash. See ARCHITECTURE.md's fau-backup section for the subvolume layout -- GRUB's own handling of a root that lives inside a named subvolume was an open assumption until `scripts/test-install.sh` confirmed it with a real boot (it needed `root=<device>`, not `root=UUID=`, to actually work) |

## Branding (not part of the minimal-base philosophy, added deliberately anyway)

| Package | Reason |
|---|---|
| libgcc | C++ exception-handling runtime (libgcc_s.so.1) needed by fastfetch (below) and, in general, anything else that links against it -- genuinely base-system infrastructure, so it stays bootstrapped (merged into the rootfs, `fau bootstrap libgcc`), not installed as an app. Not a declared dependency of fastfetch itself -- Arch/Artix assume it's always present as a base-system package, which is exactly why it's handled here instead |
| fastfetch | requested identity/branding touch — not needed to run the system, so unlike libgcc it's installed as an isolated app (`fau install fastfetch`, into root's own `~/apps/fastfetch/`, see build-rootfs.sh) rather than bootstrapped into the system root. Still finds libgcc_s.so.1 fine despite the isolation -- the app wrapper's LD_LIBRARY_PATH is additive, not exclusive, so it falls back to the dynamic linker's normal system-wide search. Shown at login via /etc/profile, configured with a custom logo (assets/floraos-logo.txt) and a Packages line reading fau's own list instead of a package manager it doesn't know about; its config and logo live inside its own app directory (`~/apps/fastfetch/config/fastfetch/`), not /etc, so `fau remove fastfetch` deletes all of it. Fetched via fau's alpm (Arch/Artix repo) fallback, not built from source (small, and not core to the OS). Resolving it pulls in Arch's full dependency closure (glibc, filesystem, tzdata, iana-etc, linux-api-headers) -- `fau` now skips anything it already built itself and strips `etc/`+`usr/include` from the rest (see ARCHITECTURE.md's fau section for why) |

## Build-host tooling (not part of FloraOS, not built from source)

| Tool | Reason |
|---|---|
| grub-mkrescue + xorriso | assembles the hybrid BIOS+UEFI bootable ISO. GRUB's own boot images (i386-pc, x86_64-efi) are embedded straight into the ISO's boot catalog by the build host's GRUB install — this runs *before* FloraOS's kernel starts, so it isn't a FloraOS package any more than the host's gcc is. Dropped a from-source syslinux+grub build (would've meant compiling 16-bit real-mode boot code) with no loss of functionality: grub-mkrescue alone covers both boot paths. |
| grub (the package, on an *installed* system only) | not shipped in the live ISO at all — `florainstall` (see above) fetches it via `fau`'s own alpm fallback straight into the target disk at install time, the same way `fau install <wm>` fetches wlroots/sway precompiled rather than building them from source. Keeps the live ISO's own footprint unchanged; only a persistent disk install ever needs a real bootloader binary of its own |
| btrfs-progs (mkfs.btrfs, on the live system, and carried onto every install) | not shipped in the live ISO either — `florainstall` fetches it via `fau`'s own alpm fallback at install time, onto the *running live system itself* (`fau bootstrap btrfs-progs` with FAU_ROOT left at its default of "/"), not the target directly: mkfs.btrfs has to actually format the target partition before anything is mounted on it, so it can't wait for a chroot the way grub-install does. That live-system fetch runs before florainstall's own `rsync` of "/" onto the target, though, so the target ends up with btrfs-progs too, as an ordinary side effect of copying the live system's own already-merged `/usr` — not a separate fetch (confirmed by reading florainstall.c's actual call order, not assumed). This is what lets `fau backup` (see ARCHITECTURE.md) run `btrfs subvolume`/`btrfs property` on an installed system after boot. Same reasoning as e2fsprogs not being needed for this any more: florainstall formats the target disk as btrfs, not ext4 |

## Explicitly excluded (and why)

| Excluded | Reason |
|---|---|
| systemd | violates the OpenRC-only constraint |
| pacman | would mean vendoring a third-party package manager's codebase; fau is written from scratch instead |
| openssh | keeps default attack surface minimal; add as an optional package later if needed |
| any GUI/desktop/browser | still not shipped in the base image — base stays headless by default. As of eudev+floraseat+linux-lts's simpledrm built-ins (see above), the *prerequisites* for one now exist; a WM/DE itself is still purely opt-in via `fau install <wm>`, matching the same isolated-app model as any other user app |
| syslog daemon | not yet scripted — see docs/TODO.md |
